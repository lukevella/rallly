Poll admin — one stage at a time

Poll management moves out of the session-identified /poll/{id} page into a dedicated, login-required page in the dashboard shell. The page is a progression, not a dashboard: after creation you invite; once responses arrive you review and pick a winner; scheduling produces the event. One stage on screen at a time, freely navigable backwards, with lifecycle controls (close, cancel, delete) always at hand.

Rev 4 changes: the Invite stage is now designed in depth — its three moments (empty, invited, responses arriving), the reminder system (individual + bulk) with anti-spam guardrails, and the response deadline surfaced in the progress card and header. Link respondents now join the invited table directly, appearing as Responded on arrival — the separate count is gone.

01

The progression model

Three stages that mirror the organizer's actual job. The stepper is both progress indicator and navigation — every stage is reachable at all times, but the page always lands on the stage the poll is in. Nothing else shares the screen with the current stage.

Annotations

  1. 1

    The deadline lives in the header as a countdown chip — visible from every stage, next to status where "how long do I have" belongs. Its home with context and actions is the Invite stage's progress card (section 02).

  2. 2

    Manage menu = everything that isn't the current step: edit details, options, settings · pause/reopen, cancel poll, delete · duplicate (Pro), export. Lifecycle is never more than two clicks away, from any stage.

  3. 3

    The stepper carries live state, so it's also the summary: Invite shows response progress, Review shows the current leader, Event shows scheduled-or-not. Clicking any step navigates — going back to Invite at any time is a first-class path, not an escape hatch.

  4. 4

    Landing rule: no responses → Invite; responses and still open → Review; finalized → Event. The page decides where you are in the job; the stepper lets you disagree.

Annotations

  1. 1

    A · Nothing competes with the invite moment. Link primary and free; email invites a collapsed Pro row (Unified Invite Dialog sections, rendered inline). The deadline is already present in the hero copy and header chip — it frames the whole campaign.

  2. 2

    A → B: the empty state promises the table. "People you invite — and anyone who responds to your link — will appear here" teaches that there is one list and one door.

  3. 3

    B · The progress card is the deadline's home. Progress and deadline are the same tension — who's in before time runs out — so they share one card: count, bar, close date. The header chip echoes it everywhere; it turns bold (≤2 days) as urgency rises.

  4. 4

    The system does the last chase: with a deadline set, one auto-reminder goes to pending invitees 24h before close — shown as a scheduled fact the host can edit or turn off. Knowing it's coming is itself an anti-spam measure: the host doesn't need to hammer manually.

  5. 5

    B · Reminders are disabled for the first 24h after inviting — the invite email is the first touch. The bulk button sits visibly disabled with "available tomorrow" on hover, so the capability is discovered before it's needed.

  6. 6

    C · One denominator: everyone we know about. 17 invitees + 9 link joiners = 26 people, 21 responded. No separate link count — a link respondent is just a row that arrives already Responded.

  7. 7

    Bulk remind targets "pending", never "all": the button is scoped and counted — "Remind pending · 5" — and opens the confirm dialog below. There is deliberately no way to email people who already responded from this screen.

  8. 8

    Link respondents join the same table, marked only by a subtle "Joined via link" secondary line. Row actions differ where reality differs: no invite to resend or revoke; remind only possible if they left an email (require-email setting). Overflow for responded rows: View votes · Edit votes · Remove.

  9. 9

    Per-row remind shows its own history: after sending, the button becomes "Reminded 2h ago" and stays disabled for 24h. The row always shows the last touch (invited / reminded), so the host can see pressure already applied before adding more.

03

Reminders — easy to send, hard to spam with

One confirm dialog for bulk; per-row buttons for individuals. The guardrails are layered: visible history on every row, a 24h cooldown per invitee, a per-poll cap, auto-exclusion of anyone recently touched, and a send button that always states the exact number of emails.

The guardrail stack 5

Quiet period — no reminders within 24h of the invite email itself
Cooldown — max one reminder per person per 24h, from any trigger (row, bulk, or auto)
Cap — max 3 manual reminders per person per poll; the row action retires with "3 reminders sent"
Visible history — every row shows the last touch before you add another
Explicit count — the send button always says how many emails are about to go out
Scheduled auto-reminder — one system nudge 24h before the deadline, announced in the progress card so the host doesn't duplicate it

Annotations

  1. 1

    Auto-exclusion with a reason: anyone inside the cooldown is unchecked, grayed, and explains itself. The host can't accidentally re-include them — the checkbox is locked until the cooldown ends.

  2. 2

    A personal note makes reminders feel human — it's prepended to the standard reminder email, which carries the participant's personal link and the deadline.

  3. 3

    The rule is stated where it applies, not buried in docs — hosts learn the system's pace from the dialog itself.

  4. 4

    The button is the receipt: "Send 4 reminders" — never "Send", never a number that disagrees with the list above it.

  5. 5

    Layered, not a single rule: each guardrail catches a different accident — double-click (cooldown), enthusiasm (cap), forgetfulness (history), scale surprise (count), and the deadline crunch is handled by the system nudge so it doesn't have to be handled by hammering.

Annotations

  1. 1

    The way back is always lit: the Invite step shows the pending count from anywhere. Chasing stragglers is one click away, and reminders live there — Review stays about deciding.

  2. 2

    Sort covers both mental models: most votes (default) for deciding, date for scanning the calendar shape. At 44 options sorting does the work; no search needed on the page.

  3. 3

    Participants is a drill-in, not a section: opens a side panel listing everyone with a ballot — each row expands to their full votes, with Edit votes and Remove in the row menu. The panel keeps the stage single-focus while covering scope item 2.

  4. 4

    "Pick this time" is the stage's verb — only rank 1 carries the button at rest, so the default path is one click. It opens the confirm dialog (section 05).

  5. 5

    Any option can be picked from its expanded row — deliberate small friction: choosing a non-leader is a considered act, taken while looking at who it excludes.

  6. 6

    Top 4–5 by default, rest on demand ("show all" renders the full ranked list, virtualized). Options management — add dates, prune 0-vote options — lives under Manage ▸ Edit options; removal warns when votes exist.

  7. 7

    Comments ride along collapsed-open at the bottom of Review — they're input to the decision. Delete per comment via overflow; disabling comments entirely is in Settings.

Annotations

  1. 1

    One dialog, not a wizard: the Review stage already did the choosing, so confirming carries the chosen time in the title. Picking a different time = back to the list, not steps inside a dialog.

  2. 2

    Consequence preview: who's in, who's out, before committing. "Can't make it · 2" expands to names.

  3. 3

    Pressure release, not a blocker: pending invitees surface here with an inline remind shortcut (same cooldown rules) — the chase-or-decide loop closes inside the decision moment. Notify is checked by default; recurring organizers can uncheck for retroactive scheduling.

  4. 4

    Cancel is the explicit "this isn't happening" path (Manage ▸ Cancel poll, any stage) — distinct from Pause (temporary stop, reversible) and Delete (destructive, removes the record). Cancel keeps the poll readable and optionally tells everyone.

Annotations

  1. 1

    Completed steps turn into the archive: Invite shows the final roster read-only; Review shows final results. The stepper's checkmarks tell the story at a glance.

  2. 2

    Manage menu shrinks with state. Finalized: Duplicate (Pro), Export, Delete. Live/Paused: full menu — edit details/options/settings, pause/reopen, cancel poll, delete, duplicate, export. Canceled polls look like Finalized minus the event card, with Reopen offered.

  3. 3

    Paused is a banner, not a place: it sits above the stepper on every stage and explains exactly what participants experience — including that the deadline clock is suspended. Reopen is immediate, no dialog.

  4. 4

    You can still pick a winner while paused — pausing to stop late votes and then scheduling is a natural sequence. When a deadline passes, the poll closes to votes and the page lands on Review with "Deadline passed — pick a time" (open question 3).

Built for what's next — without designing it

The staged model gives later features an obvious home instead of a fight for space: collaborators join the Invite stage (it's already the people-and-access screen), and activity becomes a global drawer available from any stage.

Also reserved, off-canvas:

Activity — a drawer from the header, not a stage: it's a log, not a step in the job.

"Shared with me" polls land in the collaborator's own dashboard poll list — no layout impact here.

Private polls change the Invite stage's composition (email form primary, no share link) — a swap inside the stage, not a new page.

The header status chip has room for a visibility label (Public / Private) later.

Why stages hold up

  1. →

    Every future feature is either about people (→ Invite stage), about deciding (→ Review stage), or about the record (→ drawer/menu). New capability lands inside an existing stage instead of adding a section to a stack — the progression never gets longer or noisier.

11

Open questions before hi-fi

  1. Is picking a winner free or Pro? Today "Schedule" triggers the paywall for free users, but the brief lists finalizing under core lifecycle and the conversion bet has moved to invites. Recommend free — it's the job to be done; gating it competes with the invite gate for the same moment of intent.
  2. Reminder cap tuning. Wireframed: 24h quiet period after invite, 24h cooldown per person, 3 manual reminders per person per poll, plus one scheduled auto-reminder before the deadline. Are these the right numbers, and does the auto-reminder count against the manual cap? Recommend it doesn't — it's the system's touch, not the host's.
  3. What happens when the deadline passes? Wireframed assumption: the poll stops accepting responses and the page lands on Review with "Deadline passed — pick a time". Alternatives: auto-pause (host reopens if needed) or a soft deadline (label only). Needs the deadline feature's own spec to confirm.
  4. Unmatched invitee edge case: an invitee who responds via the shared link under a different email produces two rows — one Pending (kept on the reminder path), one Responded. A "this is the same person" merge action on the pending row resolves it — deferred to the invites implementation spec.
  5. Link respondents without email: with "require participant email" off, link rows may have no email — they're visible but unreachable (no remind, no notify). Do we nudge hosts toward requiring email when they use invites/deadlines? Recommend a one-line hint in the invite section, not a forced setting.
  6. Cancel semantics. Wireframed: Cancel keeps the poll readable, marks it canceled on the invite page, optionally notifies. Does the existing canceled status match this, and can a canceled poll be reopened?
  7. Where do participant edits live long-term? Ballots sit behind a Participants drill-in panel on Review. If usage shows organizers frequently editing votes, it may deserve promotion — watch for it in usability.
  8. Where does the legacy /poll/{id} URL go? Owners: redirect here. Guests who managed via session: the Unified Invite project's account prompt is the on-ramp — needs a decided redirect + claim flow before this page ships.

Rev 4 — 14 Jul 2026: Invite stage designed in depth: three moments (just created / invited, waiting / responses coming in) sharing one anatomy (progress card · invite actions · people table). Reminder system: per-row + bulk-to-pending with layered guardrails (24h quiet period after invite, 24h cooldown, 3-per-poll cap, auto-exclusion with reasons, explicit send counts, scheduled auto-reminder before deadline). Deadline surfaced in the Invite progress card, echoed as a header countdown chip, authored in Edit details. Link respondents merged into the people table (arrive as Responded); the separate link count is gone.

Rev 3: staged progression (Invite → Review → Event) replaces the stacked layout; Cancel poll added as an explicit lifecycle action. Rev 2: ranked option list (scales to 40+ options); manual "add participant" removed.
