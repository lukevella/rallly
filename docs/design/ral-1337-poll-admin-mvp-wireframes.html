Poll admin — the page that turns a poll into a plan

Poll management moves out of the session-identified /poll/{id} page into a dedicated, login-required page in the dashboard shell. The organizer's job: get people in, watch results converge, finalize a time — without leaving this page. Wireframes below are structure and flow only; no visual design decisions are being made here.

Reading key Grayscale = the wireframe Blue = annotations, not UI Amber = Pro gate moment Green dot = responded status ▨ Dashed = reserved for a later project
01

Page anatomy

One page, two columns. The main column is the scheduling work (results, options, comments); the right rail is people & access (invites now; collaborators and activity later). Growth happens by stacking cards in the rail — no redesign needed.

Annotations

  1. 1

    Header: breadcrumb back to Polls, poll title, status chip (Live / Paused / Finalized). Sits in the standard dashboard page header.

  2. 2

    Two persistent actions: Invite (primary — the page's most valuable act) and a Manage menu carrying edit, lifecycle, export, duplicate, delete. "View invite page" opens the participant-facing page in a new tab.

  3. 3

    Column collapse: under ~1024px the rail folds beneath the hero zone (order: A, D, B, C, E, F). See section 08 for mobile.

  4. 4

    Future-proofing: collaborators and activity are rail cards. The invitee roster row (avatar · name · status chip · overflow menu) is the same row contract the collaborators card will reuse.

Annotations

  1. 1

    Winning option is named, not inferred. The chip states the current leader; the Finalize split-button defaults to it and drops down to pick another. Answers "which option is winning" without reading the table.

  2. 2

    Leading column highlighted in the grid itself. Columns are options, rows are participants — same mental model as today's desktop poll, minus the organizer's own voting row.

  3. 3

    Edit / remove a participant via row overflow: Edit votes (inline row edit, existing pattern) and Remove (confirm dialog). Covers scope item 2.

  4. 4

    Add participant manually — the organizer records someone's answer from a phone call or hallway chat. Kept, since the participant table is today misused for invites precisely because this need is real.

  5. 5

    Options on a live poll: ＋ opens the calendar picker (existing edit-options view) scoped to adding; ✕ on a row with votes warns "3 people voted for this option — their votes will be deleted".

  6. 6

    Comment moderation = delete, via overflow. Disabling comments entirely lives in Settings (section 07).

  7. 7

    The invite card is the demoted hero: link + roster, compact. "＋ Invite" reopens the full invite sections in a dialog — same composition as section 02.

  8. 8

    Roster row states: Responded (matched to a participant) or Pending. Pending overflow: Resend invite, Copy personal link, Revoke. Link-responders who were never invited count in the footer line, not the roster.

  1. 1

    Sorted by score, pre-selected to the leader. The organizer confirms rather than hunts. Availability bar + explicit vote counts per option.

    Finalized: the event replaces availability as the hero. Links out to the existing event page; the poll page remains the record of how the time was chosen.

    Manage menu shrinks with state. Finalized: Duplicate (Pro), Export, Delete — no close/reopen, no edit options. Paused: full menu minus nothing. Live: everything (see 03).

  5. 5

    Finalize still available while paused — pausing to stop late votes and then finalizing is a natural sequence.

09

Built for what's next — without designing it

The rail is an append-only stack of people-and-access cards. Collaborators and activity slot in beneath the invite card using the same row contract the roster already establishes (avatar · identity · status chip · overflow menu). Nothing in the main column moves when they arrive.

Invite

Responded
Collaborators (Pro)

Same row anatomy: avatar · name · Pending/Active chip · overflow (resend, revoke). Poll-level, capped per poll, occupies no seat.

Activity

Timeline card, read-only. Lowest priority position; can paginate internally.

Also reserved, off-canvas:

"Shared with me" polls land in the collaborator's own dashboard poll list — no layout impact here.

Private polls change the invite card's composition (email form primary, no share link) — a card-internal swap, not a page change.

The status chip in the page header has room for a visibility label (Public / Private) later.

Why a rail, not tabs

  1. →

    Tabs would hide the roster while the organizer reads results — but "who hasn't answered" and "how are results converging" are the same glance. The rail keeps both visible and gives later features a home that doesn't cost a redesign.

10

Open questions before hi-fi

  1. Is finalize free or Pro? Today "Schedule" triggers the paywall for free users, but the brief lists finalizing under core lifecycle and the conversion bet has moved to invites. Recommend free — finalize is the job to be done; gating it competes with the invite gate for the same moment of intent.
  2. Can a finalized poll be un-finalized? The wireframes treat it as terminal (delete the event to undo). If reopening must exist, it belongs in the Manage menu with a strong confirm.
  3. Notify participants when options change? Adding/removing options on a live poll may deserve an optional "let participants know" nudge. Not wireframed; needs a product call.
  4. Roster matching edge case: an invitee who responds via the shared link under a different email shows as Pending forever. Manual "mark as responded" / link-to-participant on the roster row could resolve it — deferred to the invites implementation spec.
  5. Where does the legacy /poll/{id} URL go? Owners: redirect here. Guests who managed via session: the Unified Invite project's account prompt is the on-ramp — needs a decided redirect + claim flow before this page ships.
  6. Add participant vs invite: keeping manual "＋ Add participant" (free) alongside email invites (Pro) risks the old confusion. Mitigation wireframed: they live in different sections with distinct verbs. Validate in usability pass.
